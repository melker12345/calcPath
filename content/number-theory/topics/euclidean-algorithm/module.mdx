# Euclidean Algorithm

The Euclidean algorithm computes the greatest common divisor of two integers efficiently by repeated division and taking remainders. It is one of the oldest algorithms still in use, dating back to Euclid's Elements (~300 BC), and remains the basis for almost all modern gcd computations, extended gcd for finding Bézout coefficients, and many cryptographic protocols (RSA key generation, modular inverses).

Its efficiency comes from the fact that the numbers shrink rapidly (at least by half every two steps in the worst case, Fibonacci numbers give the worst case). The algorithm also yields the extended form that expresses gcd as integer linear combination ax + by.

Mastering it is essential before studying modular arithmetic, continued fractions, and Diophantine equations.

## The Division Algorithm and GCD Definition

<!-- section: division-algorithm -->

For any integers a and b with b > 0 there exist unique integers q (quotient) and r (remainder) such that
\[
a = b q + r, \quad 0 \leq r < b.
\]

The **greatest common divisor** $\gcd(a,b)$ is the largest positive integer d that divides both a and b. We can always assume a,b ≥ 0 by signs (gcd is positive).

Key property: $\gcd(a,b) = \gcd(b, r)$ where r = a mod b. This is because any common divisor of a and b also divides any linear combination, in particular a - b q = r, and vice versa.

**ELI5**

- When you divide cookies (a) into groups of size b, the leftover r is what you have after as many full groups as possible. The common "group size" that works for both the original and the leftovers is the same.
- The gcd doesn't change if you replace the larger number by the remainder after subtracting multiples of the smaller.

**Worked Example:**

**Division algorithm on 1071 and 462**

1071 = 2 * 462 + 147   (r=147)

462 = 3 * 147 + 21

147 = 7 * 21 + 0

When remainder hits 0, the last nonzero remainder is the gcd.

## The Euclidean Algorithm

<!-- section: euclidean-algorithm-proper -->

The algorithm:

To compute gcd(a,b) assume a ≥ b > 0:

While b ≠ 0:

  Replace (a,b) by (b, a mod b)

Return a (the last nonzero remainder).

It terminates because remainders form a strictly decreasing sequence of nonnegative integers.

The number of steps is O(log min(a,b)), very fast even for huge numbers (hundreds of digits).

**ELI5**

- Repeatedly replace the big number by the leftover when divided by the small one, then swap roles. Eventually the remainder becomes zero and the other number is the gcd.
- Like the game of subtracting multiples until nothing is left; the last pile size that fits evenly is the answer.

**Worked Example:**

**gcd(48, 18)**

48 = 2*18 + 12

18 = 1*12 + 6

12 = 2*6 + 0

gcd = 6.

(Only 3 steps.)

## The Extended Euclidean Algorithm

<!-- section: extended-euclid -->

By working backwards through the remainders, we can express the gcd as ax + by for some integers x,y (Bézout's identity). This is always possible when d=gcd(a,b).

The extended version augments each step with coefficients:

Keep track of how each remainder is a combination of original a and b.

When remainder 0 appears, the previous line gives the coefficients.

This is crucial for finding modular inverses: if gcd(a,m)=1 then there is x with a x ≡ 1 mod m, namely x from the extended gcd.

**ELI5**

- Not only do we find the size of the greatest common "measure", but we also discover exactly how many of each original quantity to combine (positive and negative) to obtain exactly that measure.
- It's like finding the recipe: "3 of the first ingredient minus 1 of the second gives the gcd."

**Worked Example:**

**Extended gcd for 48 and 18 (back-substitution)**

From the steps:

6 = 18 - 1 * 12

But 12 = 48 - 2 * 18

Substitute: 6 = 18 - 1*(48 - 2*18) = 18 -1*48 +2*18 = 3*18 -1*48

Thus 6 = (-1)*48 + 3*18. So x=-1, y=3.

Verify: -48 + 3*18 = -48 + 54 = 6. Yes.

## Applications and Complexity

<!-- section: applications-complexity -->

The extended gcd immediately gives solutions to linear Diophantine equations ax + by = c when d|gcd divides c (scale the coefficients).

It is used in RSA to compute the private exponent d from e and φ(n).

For Fibonacci numbers F_{n+1}, F_n the algorithm takes the maximum number of steps (n-1 divisions), showing the worst-case bound is tight.

Modern implementations use binary gcd (Stein's algorithm) avoiding divisions for hardware, but the classic version is simplest to understand and prove correct.

**ELI5**

- Once you can write the gcd as combination, you can solve "make exactly c using a and b amounts" whenever possible.
- The algorithm is so fast that even 1000-digit numbers are handled in milliseconds; the steps grow only with the number of digits, not the value itself.

**Worked Example:**

**Solve 30x + 18y = 6**

First gcd(30,18)=6.

Extended: similar back-sub yields 6 = 1*30 + (-1)*18 ? Let's compute properly:

30=1*18 +12

18=1*12 +6

12=2*6 +0

6=18-1*12

12=30-1*18

6=18-1*(30-18)= 2*18 -1*30

So 6 = -1 *30 + 2*18

Thus x=-1, y=2 works: -30 + 36=6.

General solution: x = -1 + 3 t , y=2 -5 t  (since divide by gcd=6, add b/d=3, subtract a/d=5).

## Common Mistakes

- **Stopping too early**: The gcd is the last *nonzero* remainder, not the first remainder or the quotient.
- **Forgetting signs in extended**: x or y can be negative; that is perfectly valid and often necessary.
- **Assuming it only works for positives**: The algorithm extends to negatives by taking absolutes at the end for the gcd value; coefficients handle signs naturally.
- **Confusing with subtraction version**: Repeated subtraction works but is exponentially slower; always use modulo/remainder for efficiency.
- **When gcd does not divide c**: No integer solutions exist for ax+by=c; many students forget to check the divisibility condition before claiming a solution.
