# Kolmogorov complexity

This advanced treatment of Kolmogorov complexity (K-complexity) focuses on its mathematical properties, fundamental limitations, and deep connections to induction, learning, and the foundations of mathematics. While a basic introduction appears earlier, here we examine why K is uncomputable, what that implies for science and AI, and how it formalizes Occam's razor and randomness for individual objects.

## Properties of Kolmogorov Complexity

<!-- section: kc-properties -->

K satisfies the chain rule (up to additive constants): K(x,y) = K(x) + K(y | x*) + O(1) where x* is a shortest program for x. This is the algorithmic analogue of H(X,Y) = H(X) + H(Y|X).

K is 'almost' subadditive: K(x,y) ≤ K(x) + K(y) + O(1). It is also invariant under computable transformations: K(f(x)) ≤ K(x) + c_f for any computable f.

Plain Kolmogorov complexity (not prefix-free) and prefix complexity differ by at most a logarithmic term; most theorems hold for both.

**ELI5**

- The shortest description of the pair (x,y) is basically 'shortest description of x' plus 'shortest description of y given that you already have the shortest description of x'.
- Any computable processing of x cannot increase its complexity by more than the size of the processing program.

**Worked Example:**

K(π digits up to n) ≤ K(n) + c (the program 'compute first n digits of π' plus the bits of n). Meanwhile a random string r of length n has K(r) ≈ n + K(n) in the typical case; you cannot compress it further.

## Uncomputability and Incompressibility

<!-- section: uncomputability -->

K(x) is upper semi-computable (we can approximate it from above by dovetailing all programs) but not computable. If K were computable we could solve the halting problem: run the candidate program for |p| steps; if it hasn't halted, its output (if any) would have K at most |p| + O(1), contradicting a large enough lower bound.

A string x is c-incompressible if K(x) ≥ |x| - c. By counting, for any c the fraction of n-bit strings that are c-incompressible tends to 1 as n→∞.

Martin-Löf randomness (passing all effective statistical tests) is equivalent (up to constants) to being incompressible.

**ELI5**

- There is no algorithm that, given any x, prints the number K(x). If there were, you could detect non-halting programs by checking whether their 'would-be' output has too low a complexity.
- Most strings are incompressible; only a tiny fraction have short descriptions. 'Randomness' for a single string means 'no short program describes it'.

**Worked Example:**

Suppose K were computable. To decide if program p halts: compute k = K( output-of-p-if-it-halts ). If the machine that runs p for |p|+c steps and checks against k would contradict the definition unless p halts within that time. The contradiction shows K is uncomputable.

## Applications: Induction, Learning, and Incompleteness

<!-- section: applications-incompleteness -->

Solomonoff induction (universal prior m(x) based on K) is a formal, parameter-free embodiment of Occam's razor for sequence prediction. Its predictions converge to the truth for any computable data-generating process.

In machine learning, minimum description length (MDL) and minimum message length (MML) principles select models that minimize K(model) + K(data | model); they are practical approximations to K.

Chaitin's incompleteness: any consistent formal system of 'complexity' c can prove statements of the form 'K(x) > n' only for n bounded by c + O(1). Thus mathematics cannot prove that particular long strings are random.

Levin's universal search (Kt complexity with time-bounded K) gives an optimal (up to constants) algorithm for inverting any computable function or solving any NP search problem, given enough time.

**ELI5**

- When choosing among models that fit your data, prefer the one whose description (the model itself plus the data encoded using the model) is shortest. That is formal Occam.
- A fixed axiom system cannot certify the randomness of strings much longer than the axioms themselves; there will always be 'random' mathematical facts that the system cannot prove.

**Worked Example:**

Given data D, two models M1 (simple, fits with 100 bits of noise) and M2 (complex, fits with 10 bits of noise). The total description length K(M1) + K(D | M1) vs K(M2) + K(D | M2). MDL picks the smaller sum, automatically trading model complexity against fit.

## Common Mistakes

- **Treating K(x) as a practical compression metric for arbitrary files**: While theoretically ideal, K is uncomputable; real compressors (gzip, etc.) are computable approximations tuned to specific redundancies.
- **Forgetting the O(1) additive constants when making asymptotic claims**: Many equalities in AIT hold only up to constants; statements like 'K(x) = H(x) for typical x' are true with high probability but the deviation can be logarithmic or worse in some regimes.
- **Believing that a string with high K must 'look random' to every statistical test**: There exist strings that are incompressible yet fail some non-effective tests; the equivalence is with Martin-Löf tests (effective ones).
- **Confusing prefix-free K with plain (monotone) complexity**: The technical differences matter for the chain rule and for Ω; most high-level conclusions are the same.
- **Assuming that because a string is generated by a short program it cannot be used as a cryptographic key**: If the program is secret and the output is long, the output can still have high 'computational' entropy even if K is small.
- **Using AIT to 'prove' that science is impossible or that induction fails**: On the contrary, Solomonoff induction succeeds for all computable realities; the uncomputability is a limit on perfect prediction, not a refutation of the method.
