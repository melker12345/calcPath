# Network coding

In traditional packet networks, intermediate routers only store and forward packets. Network coding allows (or requires) routers to compute linear combinations (or other functions) of the packets they receive and forward the combinations. For multicast traffic this can achieve higher throughput than routing alone; the classic gain is demonstrated by the butterfly network.

Network coding sits at the intersection of information theory, graph theory, and algebra; random linear network coding makes it practical in large networks.

## Store-and-Forward vs Network Coding

<!-- section: store-forward-vs-nc -->

Under pure routing, each edge carries a copy of some source packet (or a duplicate). The max-flow min-cut theorem gives the maximum rate from one source to one sink, but for multiple sinks the routing rate may be strictly less than the cut value.

With network coding, an edge can carry a function (typically linear combination over a finite field) of several packets. This 'mixing' lets information flow more efficiently to multiple destinations that need different subsets of the data.

**ELI5**

- Routing is like delivering whole letters or whole boxes; each truck on a road carries one intact box.
- Network coding is like allowing the post office to open the boxes, mix the contents (say take the XOR of two letters), and send the mixture onward. The recipients, having side information, can still recover everything.

**Worked Example:**

In a network where two sources S1 S2 want to reach two sinks T1 T2, and the only paths cross on a bottleneck edge of capacity 1, routing can deliver at most rate 1 total (one source or the other, or half each), while coding on the bottleneck can deliver rate 1 to each sink (see butterfly).

## The Butterfly Network Example

<!-- section: butterfly -->

The canonical butterfly: sources S1 and S2 each produce one packet (say bits a and b). Two edge-disjoint paths from S1 to T1 and from S2 to T2, plus a cross path through a central node C that has only unit capacity to both T1 and T2.

With routing, C can forward either a or b but not both in one use; at least one sink gets only rate 1/2 or one message is lost.

With network coding, C forwards a+b (over GF(2)). T1 receives a (direct) and a+b, recovers b = a + (a+b). T2 symmetrically recovers a. Both sinks get both packets at rate 1 each; the bottleneck carried 'one packet worth' of information that served both.

**ELI5**

- The middle link sends a 'mixture' a+b. The guy who already has a can subtract to get b; the guy who has b can subtract to get a.
- One transmission on the scarce middle link satisfies the missing piece for two different people.

**Worked Example:**

Packets a from S1, b from S2. C computes and sends a XOR b. T1 has direct a, receives a⊕b → computes b. T2 has direct b, receives a⊕b → computes a. Total throughput 2 packets across a min-cut of 2 (but routing would need 3 edge uses or drop rate).

## Linear Network Coding and Throughput

<!-- section: linear-nc -->

For multicast (one source, many sinks wanting the same data), linear network coding over a sufficiently large field achieves the max-flow min-cut rate to every sink simultaneously — something routing cannot always do.

Each edge carries a linear combination of the source packets; the coefficients can be chosen centrally (algebraic NC) or randomly (random linear NC, with high probability invertible at the sinks when field is large).

Decoding at a sink reduces to solving a linear system whose matrix is the product of the local coding matrices along the paths; the sink needs the matrix to be full rank.

**ELI5**

- Each packet on the wire is a 'recipe' — a vector of coefficients telling how much of each original source packet is mixed in.
- At the destination you collect enough independent recipes; linear algebra (Gaussian elimination) recovers the original ingredients.

**Worked Example:**

Source has k packets (a vector in F_q^k). Each internal node chooses random coefficients and outputs the corresponding linear combo. After receiving k independent combinations (the global encoding vectors form an invertible matrix), the sink inverts the matrix and obtains the original k packets. For q large enough or with high probability for random choice, success probability →1.

## Common Mistakes

- **Thinking network coding always doubles throughput**: The gain is topology-dependent; in some networks (trees) coding gives no gain over routing; the butterfly is the smallest example with gain 2.
- **Confusing network coding with source or channel coding**: Network coding operates on packets at intermediate nodes; it is orthogonal and can be composed with end-to-end source/channel coding.
- **Assuming the field size can be arbitrarily small**: For linear NC to achieve capacity, the field must be large enough relative to the number of sinks or edges; GF(2) sometimes works but not always.
- **Forgetting header overhead for coefficients**: In random linear NC the coding vector (coefficients) must be sent with the packet or known by other means; this overhead vanishes asymptotically but matters for small generations.
- **Believing coding is only for multicast**: While the big theorem is for multicast, network coding ideas (including index coding, functional compression, compute-forward) apply to many other traffic patterns and have practical uses in wireless (COPE, MORE) and distributed storage.
- **Ignoring that practical deployments use generation-based or sliding-window NC rather than the block model of the theorems**: Real systems chunk data into generations to keep matrices small and decoding delay low.
