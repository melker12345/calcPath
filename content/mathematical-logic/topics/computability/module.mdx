# Computability Theory

Computability theory (recursion theory) asks: which functions and problems can be solved by a purely mechanical procedure that always terminates with the correct answer? It provides the mathematical foundation for the notion of "algorithm" and reveals fundamental limits of computation.

The central models are Turing machines (Alan Turing 1936) and the equivalent recursive functions (Church, Kleene). They turned out to capture exactly the intuitive idea of effective calculability.

## Turing Machines and Algorithms

<!-- section: turing -->

Turing machines provide the precise mathematical model of what it means for a procedure to be "mechanical" and effective. Any question about whether a problem is algorithmically solvable is ultimately a question about what Turing machines (or equivalent formalisms) can and cannot do.

A Turing machine has an infinite tape, a finite set of internal states, a read/write head, and a finite table of instructions. Despite the extreme simplicity of the device, Turing showed that it can simulate any other reasonable model of computation (lambda calculus, register machines, modern programming languages). The functions it can compute are called the computable (or recursive) functions.

In this section you will learn the components of a Turing machine, the definition of computability, several examples of computable and non-computable functions, and why the model is robust under changes of formalism.

A **Turing machine** consists of:

- A finite set of states, including a start state and halting states.
- An infinite tape divided into cells, each holding a symbol from a finite alphabet (usually including blank).
- A head that reads/writes one cell and moves left or right.
- A transition table: given current state and symbol read, what to write, which direction to move, and next state.

A function $f:\mathbb{N}\to\mathbb{N}$ is **computable** (Turing-computable, recursive) if there is a TM that, started with input $n$ (in unary or binary), eventually halts with $f(n)$ on the tape.

**ELI5**

- Imagine an infinitely long strip of paper, a read/write head, and a little program (the finite state table) that tells the head "if you see a 1 and you are in state q3, write 0, move right, go to state q7".
- Anything a modern computer can do (given enough time and memory) can be simulated by such a simple device. Conversely, if a TM cannot compute it, no real computer program can either.

**Worked Example:**

A TM that computes the successor function on unary numbers (tally marks): scan right until you find the first blank, write a 1, halt. This works for any input length.

## Recursive Functions and Church-Turing Thesis

<!-- section: recursive -->

The class of **recursive functions** is the smallest class containing:

- Zero function, successor, projection functions (base).
- Closed under composition, primitive recursion, and the $\mu$-operator (unbounded search / minimization).

**Church-Turing thesis** (not a theorem): a function is effectively calculable by an algorithm if and only if it is recursive / Turing-computable. All reasonable models of computation (lambda calculus, register machines, etc.) define the same class.

**ELI5**

- You start with the most basic operations (copy a number, add one, pick out an argument) and are allowed to build bigger programs by plugging them together, repeating a number of times given by another number, or searching for the first number that makes a condition true.
- The thesis says this captures everything humans would call "a definite mechanical procedure".

**Worked Example:**

Addition is primitive recursive: $\text{add}(x,0) = x$, $\text{add}(x,S(y)) = S(\text{add}(x,y))$.

Multiplication similarly from addition. Exponentiation from multiplication. The Ackermann function is a famous example that grows faster than any primitive recursive function and requires the full $\mu$-operator (general recursion).

## The Halting Problem and Undecidability

<!-- section: halting -->

The **halting problem**: given (description of) a Turing machine $M$ and input $w$, does $M$ halt on $w$?

**Theorem** (Turing 1936): the halting problem is undecidable.

Proof by contradiction: assume a TM $H$ decides it. Build a diagonal machine $D$ that on input $M$ runs $H(M,M)$; if $H$ says $M$ halts on itself then $D$ loops, else halts. Then ask what $D$ does on input $D$: contradiction either way.

Many other problems reduce to halting and are therefore undecidable: "does this program ever print 0?", "does this program halt on empty input?", validity in FOL, etc.

**ELI5**

- There is no general-purpose debugger that, given any program and any input, can always tell you in finite time whether that program will eventually stop or run forever on that input.
- The proof is by diagonalization — the same technique used to show the reals are uncountable.

**Worked Example:**

The set of (codes of) Turing machines that halt on the empty tape is undecidable (special case of halting). If it were decidable we could solve the full halting problem by modifying the machine to ignore its input and run the original on empty.

## Rice's Theorem and Decidable vs. Undecidable Properties

<!-- section: rice -->

**Rice's theorem**: Any non-trivial property of the language recognized (or function computed) by Turing machines is undecidable.

"Non-trivial" means: there exists at least one TM with the property and at least one without.

Examples of undecidable properties: "does the TM accept the empty string?", "does it accept any string?", "is the language finite?", "is the function computed total?", "does it ever output 42?".

Properties that *are* decidable are usually syntactic (about the code of the machine itself, not its behavior).

**ELI5**

- If you ask a question that is really about *what the program does* (its input-output behavior) rather than how it is written on the page, and the question is interesting (sometimes yes, sometimes no), then there is no algorithm that answers it correctly for every possible program.
- "Does this program contain the word 'while'?" is syntactic and decidable. "Does this program contain an infinite loop?" is behavioral and undecidable.

**Worked Example:**

"Does the TM accept at least one string?" is non-trivial (some accept everything, some accept nothing) hence undecidable by Rice.

## Common Mistakes

- **Human inspection is not a decider:** Believing "my program always halts because I can't think of an input that makes it loop" is wrong. Human inspection is not a decision procedure that works for all programs.

- **Undecidable versus uncomputable in general:** Confusing "undecidable" with "impossible to compute anything" is mistaken. Plenty of useful functions are computable; only certain questions about arbitrary programs are not.

- **Heuristics still exist for undecidable problems:** Thinking that because a problem is undecidable there is no useful approximation or heuristic is false. Compilers, antivirus, and static analyzers use sound but incomplete methods all the time.

- **Recursive versus primitive recursive:** Equating "recursive" with "primitive recursive" is incorrect. The latter is a proper subclass (Ackermann is total recursive but not primitive recursive).

- **Halting says nothing about speed:** Assuming that if a TM halts on all inputs it must do so "quickly" is wrong. Halting says nothing about time or space complexity.

- **Infinity is not the halting issue:** Claiming the halting problem is undecidable "because there are infinitely many programs" misses the point. Infinity is not the issue; diagonalization against the assumed decider is.
